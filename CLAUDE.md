## The 4-Agent Architecture

```
┌─────────────────────────────────────────────────────────┐
│                    ORCHESTRATOR (main.py)                │
│  Controls flow, passes state, handles human checkpoint   │
└──────────┬──────────────────────────────────────────────┘
           ▼
┌─────────────────────┐
│   AGENT 1           │  Tools: web_search, scrape_page, save_jd
│   Job Scraper       │  Input:  search query string
│                     │  Output: list of saved JD file paths
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│   AGENT 2           │  Tools: read_file, read_profile
│   JD Analyst        │  Input:  JD file paths
│                     │  Output: ranked + scored JD list (JSON)
└──────────┬──────────┘
           ▼
    ┌─────────────┐
    │  👤 HUMAN   │  Human reviews scores, approves shortlist
    │  CHECKPOINT │  Input: "approve [1,3,5]" or "approve all"
    └──────┬──────┘
           ▼
┌─────────────────────┐
│   AGENT 3           │  Tools: read_file, read_profile, write_gap_report
│   Gap Analyser      │  Input:  approved JD paths
│                     │  Output: gap_report.md + gap_report.json
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│   AGENT 4           │  Tools: read_file, append_to_study_plan
│   Study Updater     │  Input:  gap_report.json
│                     │  Output: study_plan.md updated with new learning items
└─────────────────────┘
```

## Shared State Pattern

A plain Python dict passed between agents in `main.py`. **Do not over-engineer this in Phase 1** — LangGraph replaces it in Phase 2.

```python
state = {
    "search_query": "Senior AI Engineer UK remote",
    "jd_paths":        [],   # filled by Agent 1 — paths to saved JD files
    "scored_jds":      [],   # filled by Agent 2 — list of {path, score, reasoning}
    "approved_paths":  [],   # filled by human checkpoint
    "gap_report_path": "",   # filled by Agent 3
}
```

## File Format Decisions

| File | Format | Reason |
|---|---|---|
| `my_profile.md` | Markdown | Human-readable reference, stable |
| `JD/*.md` | Markdown | For Human to read and review |
| `JD/*.json` | JSON | For agents to process — Agent 1 saves both |
| Agent 2 scored output | JSON | Machine-consumed by Agent 3 |
| `gap_report.md` | Markdown | Human-readable gap summary |
| `gap_report.json` | JSON | Structured `gaps[]` array consumed by Agent 4 |
| `study_plan.md` | Markdown | Human-edited, not machine-parsed |

**Key rules:**
- Pass only the relevant section of a file to an agent, not the whole file (token efficiency)
- Every JD JSON must have the same fields — consistent structure = reliable extraction
- Agent 1 normalises raw JD text into clean structured JSON on save

### JD JSON Schema

```json
{
  "id": "JD1",
  "title": "Senior AI Engineer",
  "company": "PropTech UK",
  "rate": "£900/day",
  "contract_type": "Outside IR35",
  "required": ["LangGraph", "AWS Lambda", "TypeScript", "10+ years SE"],
  "nice_to_have": ["MLOps", "NLP", "microservices"],
  "stack": ["Python", "TypeScript", ".NET", "AWS"],
  "fit_score": null
}
```

## Key Concepts Being Learned in This Phase

| Concept | Where it appears |
|---|---|
| ReAct loop (Reason → Act → Observe) | Every agent call |
| Tool use / function calling | All 4 agents |
| Structured outputs (JSON) | Agent 2 scoring, Agent 3 gap report |
| File I/O as agent tools | Agents 1, 3, 4 |
| Human-in-the-loop | Checkpoint after Agent 2 |
| Agent chaining / pipeline | Orchestrator in `main.py` |
| State management (manual dict) | Shared `state` in `main.py` |
| Prompt engineering | Each agent's system prompt |
| Token efficiency | Passing only relevant file sections |
